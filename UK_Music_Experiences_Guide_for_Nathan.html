2. What Is a Website, Really?

Before we talk about how we built yours, it helps to understand what a website actually is. Don't worry — we'll keep this simple.

The Three Ingredients of Any Website

A website is like a concert. You need three things: the music itself (the files), a venue to perform it in (a server/host), and an address so people can find the venue (a domain name).
  1. The Files — Every website is made up of files, just like documents on your computer. These files contain the text, images, layout instructions, and interactive behaviour of your website. When someone visits your site, these files get sent to their browser (Chrome, Safari, Firefox, etc.), which then displays them as the page you see.
  2. A Host (Server) — Your files need to live somewhere that's accessible to the internet 24/7. This is called "hosting". Think of it as renting a small space on a computer that's always turned on and connected to the internet. When someone types in your web address, they're actually connecting to that computer, which sends them your files.
  3. A Domain Name — This is the web address people type in, like ukmusicexperiences.co.uk. Without it, your site would only be reachable via a long number (like a phone number). A domain name is the human-friendly label.

Two Types of Websites

This is an important distinction that affects cost, complexity, and what's possible:

Static Website

Like a printed brochure — the content is pre-made and the same for everyone. The browser does all the work.

Pros: Fast, cheap/free to host, very secure, simple.

Cons: Updating content requires someone with a bit of technical knowledge (or a build step).

Our prototype is this type.

Dynamic Website

Like a live performance — the server builds each page on the fly, often pulling from a database.

Pros: Easy content management (like WordPress), user accounts, real-time features.

Cons: Needs a server running 24/7 (costs more), more complex, potential security issues.

WordPress sites are this type.

A static website is like a CD — the music is already recorded and plays the same every time. A dynamic website is like a live band — it creates the experience fresh each time someone visits, which is more flexible but requires more infrastructure.

3. Our Prototype: How It Works

We built a working prototype to show what's possible. Here's what it does:

The Interactive Map

The centrepiece of the website is a full interactive map of the UK, powered by OpenStreetMap (a free, open-source mapping service — think of it as the Wikipedia of maps). Every music opportunity from your spreadsheet appears as a pin on the map.

Each type of opportunity has its own unique icon:

  • Classes, Workshops, Ensembles, Projects
  • Instrument Lessons, Mentoring, Music Centres
  • Performance Opportunities, Grants/Funds
  • Networks, Radio, Work Experience

When you click on a pin, a popup appears showing all the details: the opportunity name, type, location, genre, age range, cost, and a link to the organisation's website.

The Search & Filter System

Above the map, there are search controls that let visitors:

  • Search by name — type any word and it filters instantly
  • Filter by type — e.g., show only "Workshops" or only "Ensembles"
  • Filter by region — e.g., show only "South West" or "Scotland"
  • Search by proximity — type an address (e.g., "Bristol") and a radius (e.g., 25 km), and it shows only opportunities within that distance, sorted by how close they are

The Table View

Next to the map, there's a table listing all the matching opportunities. The table and map are linked — hovering over a row in the table highlights the corresponding pin on the map, and clicking a row makes the map fly to that location.

How Many Opportunities?

The prototype currently displays 918 music opportunities across the UK, all loaded from your original spreadsheet data.

4. How We Built It

This section explains the technology we used. Don't worry about memorising the names — the point is to show you that everything we used is free and well-established.

Step 1: The Foundation — React + TypeScript

If building a website is like writing a piece of music, then React is like your notation software (Sibelius, Finale). It's a tool that helps you organise and structure your work efficiently. TypeScript is like having a proofreader who catches wrong notes before you perform.

React is a free tool made by Meta (Facebook) that helps developers build interactive websites. It's used by millions of websites worldwide, including Instagram, Netflix, and Airbnb. It's the industry standard for building modern web interfaces.

TypeScript is an extension of JavaScript (the programming language that makes websites interactive) that helps catch errors before they cause problems. Think of it as spell-check for code.

Vite (pronounced "veet") is the build tool that packages everything together. It takes all our code and turns it into the files that a web browser can understand.

Step 2: The Map — Leaflet + OpenStreetMap

Leaflet is a free, open-source map library. It's the code that makes the map interactive — letting users zoom, pan, click on markers, and see popups. It's used by thousands of websites, including government portals, news organisations, and non-profits.

OpenStreetMap provides the actual map images (the roads, terrain, labels). It's completely free and community-maintained. We specifically use a style called "CartoDB Light" which gives a clean, light-coloured map that doesn't distract from the data pins.

No API key, no subscription, no payment needed for either of these.

Step 3: Turning Addresses into Map Pins — Geocoding

Your spreadsheet has addresses like "Bristol" or "Manchester Town Hall" — but a map needs exact coordinates (latitude and longitude numbers). The process of converting an address into coordinates is called geocoding.

We used Nominatim, which is OpenStreetMap's free geocoding service. We wrote a script that went through every unique address in your data, looked up its coordinates, and saved the results. This only needs to happen once — after that, the coordinates are stored and the map pins appear instantly.

Key point: Geocoding was the trickiest part of the build. Out of your ~918 entries that had location data in the KML file, we successfully mapped all of them. Some addresses were too vague for the geocoder to find, so in a production version, you might want to manually check the ones that got placed in the wrong spot.

Step 4: The Data Pipeline

Here's how your spreadsheet data becomes a working website:

  1. Your Google Sheet → exported as CSV (spreadsheet format) and KML (map format)
  2. Our geocoding script → reads the data, looks up coordinates for each address
  3. A JSON file → all 918 opportunities with their coordinates, stored in a structured format the website can read
  4. The website → loads this file when someone visits, displays everything on the map and in the table
It's like preparing sheet music for a concert. Your spreadsheet is the handwritten score. We transcribed it into a clean digital format (JSON), and now the website can "perform" it for anyone who visits.

Step 5: Search, Filters, and Distance

The search and filter system works entirely in the visitor's browser. When someone types a search term, the website instantly goes through all 918 opportunities and shows only the matching ones. No server needed — it's all done locally.

For the proximity search (finding opportunities near a specific location), we use a mathematical formula called the Haversine formula to calculate the distance between two points on the Earth. It's the same maths that sat-nav systems use.

7. Other Ways to Build This

Our prototype used React + Leaflet, but there are several other approaches. Each has trade-offs:

Option A: WordPress + Map Plugin

What it is: WordPress is the world's most popular website builder. It powers about 40% of all websites. You can add a map plugin to display your data.

How it works: You install WordPress (many hosting providers do this with one click), choose a theme (the visual design), and install a map plugin like "WP Go Maps" (formerly WP Google Maps) or "MapPress". You'd import your spreadsheet data through the plugin's interface.

Pros:

  • You can update content yourself without coding knowledge
  • Huge ecosystem of themes and plugins
  • Lots of people know WordPress, so easy to find help

Cons:

  • Needs traditional hosting (not free) — typically £50–100/year
  • Map plugins may have limitations (e.g., max number of pins on the free version)
  • Slower than a static site
  • Requires ongoing maintenance (WordPress updates, security patches)
  • Some map plugins require a paid Google Maps API key

Estimated cost: £50–200/year for hosting, £0–80/year for premium plugins, plus development time.

Option B: Google My Maps

What it is: Google's free tool for creating custom maps. You can import your spreadsheet directly and get a map with pins.

Pros:

  • Completely free
  • Very easy — you could do this yourself in an afternoon
  • Can be embedded in any website
  • Google handles all the technical infrastructure

Cons:

  • Limited customisation — looks like a Google product, not your brand
  • No advanced filtering or search (just basic categories)
  • Limited to 2,000 points per layer (you have ~918, so this is fine for now)
  • No table view or proximity search
  • Depends entirely on Google — if they change or discontinue the service, you're affected

Estimated cost: £0

Option C: Squarespace / Wix + Embedded Map

What it is: Website builders with drag-and-drop interfaces. You'd create a website and embed a map (either Google My Maps or a third-party map widget).

Pros:

  • Beautiful, professional-looking templates
  • No coding required — entirely visual editing
  • Built-in hosting and domain management

Cons:

  • Monthly subscription: £12–30/month (£144–360/year)
  • The map part would still be basic (an embed from Google My Maps or similar)
  • You can't build the rich interactive features we have (proximity search, linked table, type-based icons)

Estimated cost: £144–360/year

Option D: Mapbox or Google Maps Platform (Custom Development)

What it is: Professional mapping platforms with more features than OpenStreetMap, but with usage-based pricing.

Pros:

  • Very polished, professional map styles
  • Better geocoding accuracy
  • More customisation options

Cons:

  • Free tier has limits (Mapbox: 50,000 map loads/month free; Google: $200/month credit)
  • Can get expensive if the site becomes popular
  • Requires an API key and account management
  • Still needs a developer to build the website around it

Estimated cost: Potentially free for low traffic, but unpredictable costs if the site grows.

Option E: No-Code Tools (Airtable + Softr / Stacker)

What it is: Use Airtable (a fancy spreadsheet) as your database, and a tool like Softr or Stacker to automatically generate a website from it — including maps.

Pros:

  • You can update data in a spreadsheet interface
  • No coding at all
  • Map view built in

Cons:

  • Monthly costs add up: Airtable free + Softr from £40/month
  • Less customisation than a custom-built site
  • Locked into their platform

Estimated cost: £0–480/year depending on tier.

8. Keeping the Data Up to Date

One important question is: how do you add new opportunities or update existing ones?

With Our Current Approach

Right now, the data is "baked in" to the website. This means:

  1. You update your Google Sheet as usual
  2. Export it again (as CSV or KML)
  3. Someone with technical knowledge runs the geocoding script to process new entries
  4. The website is rebuilt and re-deployed (takes about 30 seconds once set up)

This could potentially be automated so that the website rebuilds itself whenever you update the spreadsheet, but that would require some additional setup.

Making It Easier

If ease of updating is important (which it probably is, since you'll be adding new opportunities), here are some options:

  • Automated pipeline: Connect Google Sheets directly to the website build process. When you update the sheet, the site rebuilds automatically. Requires initial setup by a developer, but then works hands-off.
  • Simple admin page: Build a basic form where you can add/edit opportunities through a web interface. This adds complexity but makes updates very easy.
  • WordPress approach: Use the WordPress admin panel to manage entries. Familiar interface, no coding needed for updates.

10. Glossary

Here's a reference for the technical terms used in this document:

Term What It Means
API Application Programming Interface. A way for one piece of software to talk to another. Like a waiter in a restaurant — you tell the waiter what you want, they go to the kitchen, and bring back your food. You don't need to know how the kitchen works.
Browser The program you use to visit websites: Chrome, Safari, Firefox, Edge, etc.
Build The process of converting source code into the files that a browser can understand. Like pressing "Export" on a project.
Client-side Things that happen in the visitor's browser, on their own computer. Our website is entirely client-side.
CSV Comma-Separated Values. A simple file format for spreadsheet data. When you download a Google Sheet, you can save it as CSV.
Deploy Putting the website live so people can visit it. Like publishing a track — moving it from your studio to Spotify.
Domain name The web address people type in, like bbc.co.uk or ukmusicexperiences.co.uk.
Framework A pre-built set of tools that makes building software easier. Like using a DAW (Logic, Ableton) instead of recording directly to tape.
Geocoding Converting a human-readable address (e.g., "Bristol") into map coordinates (51.4545, -2.5879) that a map can plot.
Git / GitHub A version control system and its online platform. It keeps track of every change ever made to the code, like unlimited undo. GitHub is where the code lives online.
Hosting A service that stores your website files and serves them to visitors. Like a venue for your website to "perform" in.
HTML HyperText Markup Language. The basic building blocks of every web page — defines the structure and content.
JavaScript The programming language that makes websites interactive. It runs in the browser and powers things like our search filters and map interactions.
JSON JavaScript Object Notation. A way to store structured data that computers can easily read. Think of it as a very organised spreadsheet.
KML Keyhole Markup Language. A file format for geographic data, used by Google Earth and Google Maps. Your map data was exported in this format.
Leaflet A free, open-source JavaScript library for interactive maps. The engine behind our map.
Open source Software whose code is freely available for anyone to use, modify, and share. Most of the internet runs on open-source software.
React A free tool (by Meta/Facebook) for building interactive websites. The most popular choice for modern web development.
Server A computer that's always on and connected to the internet, serving your website to visitors.
SSL Secure Sockets Layer. The technology that puts the padlock icon in your browser and makes web addresses start with "https://". Encrypts data between the visitor and the website.
Static site A website made of fixed files that don't change based on who's visiting. Fast, secure, and cheap to host.
TypeScript An improved version of JavaScript with built-in error checking. Like having spell-check for code.
Vite A build tool that packages website code for production. Fast and modern.
